# NC-263 Plan: Chat Onboarding Shell (Revised — Iteration 5)

## What changed from Iteration 4 (addressing auditor rejections)

### Auditor concern 1 — Webchat embed snippet contract mismatch

**Problem (verbatim from auditor):** The plan renders `` `<script src="https://widget.camello.io/embed.js" data-tenant-id="${tenantId}"></script>` `` and gates the entire webchat setup UI on `channelChoice === 'webchat' && tenantId`. But the actual codebase (`Step4ConnectChannel.tsx:27`) uses `${process.env.NEXT_PUBLIC_WIDGET_URL ?? 'http://localhost:5173'}/widget.js` with no `tenantId` dependency. `getStatus()` (`onboarding.ts:431–435`) does not return `tenantId`. A resumed user at step 6 has `tenantId === null` → webchat UI is permanently hidden.

**Fix 9 — Match the actual widget contract:**
- `widgetSnippet` is now a `const` computed inside the component body: `` `<script src="${process.env.NEXT_PUBLIC_WIDGET_URL ?? 'http://localhost:5173'}/widget.js" async></script>` `` — verbatim match of `Step4ConnectChannel.tsx:27`.
- Webchat setup UI condition changed from `channelChoice === 'webchat' && tenantId` → `channelChoice === 'webchat'` (no `tenantId` guard).
- `const [tenantId, setTenantId] = useState<string | null>(null)` **removed** from state.
- `setTenantId(data.tenantId)` **removed** from Effect 4's `onSuccess`.
- `setTenantId(null)` **removed** from Effect 2's reset block.
- `handleCopySnippet` now receives `widgetSnippet` (the variable) instead of an inline template literal.

### Auditor concern 2 — WhatsApp webhook configuration flow dropped

**Problem (verbatim from auditor):** The planned WhatsApp UI only collects Phone Number ID and Access Token, but NC-257 and `Step4ConnectChannel.tsx:25,157–206` require `channel.webhookConfig` read-only fields (webhook URL + verify token + instructions), which the tenant needs to complete Meta webhook registration. Without those fields, inbound WhatsApp is non-functional.

**Fix 10 — Add `channel.webhookConfig` query + read-only UI to WhatsApp section:**
- `import { Skeleton } from '@/components/ui/skeleton'` added to `ChatOnboarding.tsx` imports.
- `const webhookCfg = trpc.channel.webhookConfig.useQuery(undefined, { enabled: isLoaded && !!organization })` added to tRPC hooks section.
- `handleCopyValue` function added (same pattern as `Step4ConnectChannel.tsx:35–37`): writes to clipboard without toggling `copied` state.
- WhatsApp section in `ask_channel` JSX gains two read-only fields (after the access token input, before the save button):
  - Webhook URL: label + skeleton-or-input (readOnly) + copy button
  - Verify token: label + skeleton-or-input (readOnly, font-mono) + copy button
  - Webhook instructions paragraph
- i18n keys added: `webhookUrlLabel`, `webhookVerifyTokenLabel`, `webhookInstructions`, `copy`.

- **Test 5** (new): Webchat snippet renders when `channelChoice = 'webchat'` is selected by a user resumed to step 6. Verifies that no `tenantId` is required (organization is null, provision never fires).
- **Test 6** (new): WhatsApp webhook URL + verify token read-only fields render with data from `channel.webhookConfig` query, matching `step4-connect-channel.test.tsx:141–157`.

## Objective

Replace the 6-step stepper at `/onboarding` with a single-page chat-style UI. All existing tRPC procedures are preserved unchanged. Only the UI changes.

## State Machine

```
ChatStep type:
  'creating_org' | 'provisioning' | 'ask_description' | 'generating_agent' |
  'confirm_agent' | 'collecting_knowledge' | 'ask_website' | 'ask_channel' | 'done'
```

---

## Numeric Step Mapping (1-to-1, no shared values)

| Chat Step | Saved `onboardingStep` | Trigger |
|-----------|------------------------|---------|
| `creating_org` / `provisioning` | (not saved) | — |
| `ask_description` | 2 | `provision.mutate` → `onSuccess` |
| `confirm_agent` | 3 | `parseBusinessModel.mutate` → `onSuccess` |
| `collecting_knowledge` | 4 | `setupArtifact.mutate` → `onSuccess` |
| `ask_website` | 5 | `knowledge.ingest.mutate` → `onSuccess` |
| `ask_channel` | 6 | `handleWebsiteSubmit` (both URL submit AND skip) |
| `done` | — | `onboarding.complete()` sets `onboardingComplete: true`; `onboardingStep` integer not consulted |

**Important constraint:** `saveStep` input schema is `step: z.number().int().min(1).max(6).optional()` (verified, `onboarding.ts:443`). The highest step ever saved is 6 (`ask_channel`). The `done` state is gated by the `onboardingComplete` boolean flag, never a step integer ≥ 7. Do not attempt to save step 7.

## Helper Functions (concrete implementations)

### `STEP_ORDER` and `stepFromNumber` (module scope — no closure needed)

```tsx
const STEP_ORDER: ChatStep[] = [
  'creating_org',
  'provisioning',
  'ask_description',
  'generating_agent',
  'confirm_agent',
  'collecting_knowledge',
  'ask_website',
  'ask_channel',
  'done',
];

function stepFromNumber(n: number): ChatStep {
  if (n <= 1) return 'creating_org';
  if (n === 2) return 'ask_description';
  if (n === 3) return 'confirm_agent';
  if (n === 4) return 'collecting_knowledge';
  if (n === 5) return 'ask_website';
  if (n >= 6) return 'ask_channel';
  return 'creating_org';
}
```

### `isAtOrPastStep` (inside component — closes over `chatStep`)

Must be defined INSIDE the component body so it closes over `chatStep`. Used only in render (not in effects), so no `useCallback` is needed:

```tsx
const isAtOrPastStep = (target: ChatStep): boolean =>
  STEP_ORDER.indexOf(chatStep) >= STEP_ORDER.indexOf(target);
```

### Effects

**Effect 1 — Resume on status load (runs once via `hasResumed` ref):**


### Handlers

```tsx
// ask_description submit
const handleDescriptionSubmit = (desc: string) => {
  setBusinessDescription(desc);
  setChatStep('generating_agent');  // synchronous → typing indicator shows immediately
  parseModel.mutate(
    { description: desc, locale: locale as 'en' | 'es' },
    {
      onSuccess: (s) => {
        const sug = s as Suggestion;
        setSuggestion(sug);
        setAgentName(sug.agentName);
        setChatStep('confirm_agent');
        saveStep.mutate({ step: 3, suggestion: sug, businessDescription: desc });
      },
    },
  );
};

// confirm_agent
const handleConfirmAgent = () => {
  const finalName = isEditingName && editedName.trim() ? editedName.trim() : agentName;
  if (!suggestion) return;
  setAgentName(finalName);
  setupArtifact.mutate(
    {
      name: finalName,
      type: suggestion.agentType,
      personality: suggestion.personality,
      constraints: suggestion.constraints,
    },
    {
      onSuccess: () => {
        setChatStep('collecting_knowledge');
        saveStep.mutate({ step: 4 });
      },
    },
  );
};

// knowledge (quick facts) — saves step 5 on ingest success before transitioning
const handleKnowledgeSubmit = (facts: string) => {
  ingest.mutate(
    { content: facts, title: 'Quick facts', sourceType: 'upload' },
    {
      onSuccess: () => {
        saveStep.mutate({ step: 5 });
        setChatStep('ask_website');
      },
    },
  );
};

// website URL (optional) — ALWAYS saves step 6 (both URL submit AND skip paths).
// saveStep max is 6 (verified: onboarding.ts:443 — z.number().int().min(1).max(6)).
const handleWebsiteSubmit = (url: string) => {
  saveStep.mutate({ step: 6 });
  if (!url.trim()) {
    setChatStep('ask_channel');
    return;
  }
  queueUrl.mutate(
    { url },
    {
      onSuccess: () => setChatStep('ask_channel'),
      onError:   () => setChatStep('ask_channel'),
    },
  );
};

// WhatsApp channel save — upsertChannel → completeOnb → setChatStep('done')
const handleWhatsAppSubmit = () => {
  let valid = true;
  if (!whatsAppPhone.trim()) { setPhoneError(t('phoneRequired')); valid = false; }
  if (!whatsAppToken.trim()) { setTokenError(t('tokenRequired')); valid = false; }
  if (!valid) return;
  upsertChannel.mutate(
    { channelType: 'whatsapp', phoneNumber: whatsAppPhone, credentials: { access_token: whatsAppToken } },
    { onSuccess: () => completeOnb.mutate() },
  );
};

// Copy webchat snippet — shows "Copied!" feedback (setCopied toggles for 2s)
const handleCopySnippet = (snippet: string) => {
  void navigator.clipboard.writeText(snippet).then(() => {
    setCopied(true);
    setTimeout(() => setCopied(false), 2000);
  });
};

// Copy any single value — no "Copied!" toggle (Fix 10 — matches Step4ConnectChannel.tsx:35-37)
const handleCopyValue = (value: string) => {
  void navigator.clipboard.writeText(value);
};
```

```tsx
                </button>

        {chatStep === 'done' && (
          <ChatBubble role="bot">
            <p>{t('done', { agentName })}</p>
            <button
              onClick={() => router.push('/dashboard')}
              className="mt-4 rounded-lg bg-teal px-6 py-2 text-sm text-white">
              {t('openDashboard')}
            </button>
          </ChatBubble>
        )}

        <div ref={scrollRef} />
      </div>
    </div>
  </div>
);
```

## 4. `ChatInput.tsx`

### File header (Fix 7)

```tsx
'use client';
import { useState } from 'react';
```

Both the `'use client'` directive and the `useState` import are required. `ChatInput` uses local state for value and error tracking.

### Full implementation

```tsx
'use client';
import { useState } from 'react';

interface ChatInputProps {
  type: 'textarea' | 'text' | 'url';
  placeholder?: string;
  onSubmit: (value: string) => void;
  disabled?: boolean;
  submitLabel?: string;
  minLength?: number;
  maxLength?: number;
  skipLabel?: string;
  onSkip?: () => void;
}

export function ChatInput({
  type, placeholder, onSubmit, disabled, submitLabel,
  minLength, maxLength, skipLabel, onSkip,
}: ChatInputProps) {
  const [value, setValue] = useState('');
  const [error, setError] = useState('');

  const handleSubmit = () => {
    if (minLength && value.trim().length < minLength) {
      setError(`Minimum ${minLength} characters`);
      return;
    }
    setError('');
    onSubmit(value.trim());
  };

  return (
    <div className="flex flex-col gap-2">
      {type === 'textarea' ? (
        <textarea
          value={value}
          onChange={e => { setValue(e.target.value); setError(''); }}
          placeholder={placeholder}
          disabled={disabled}
          maxLength={maxLength}
          rows={3}
          className="w-full rounded-xl border px-4 py-3 text-sm focus:border-teal focus:outline-none focus:ring-1 focus:ring-teal disabled:opacity-50"
        />
      ) : (
        <input
          type={type === 'url' ? 'url' : 'text'}
          value={value}
          onChange={e => { setValue(e.target.value); setError(''); }}
          placeholder={placeholder}
          disabled={disabled}
          className="w-full rounded-xl border px-4 py-3 text-sm focus:border-teal focus:outline-none focus:ring-1 focus:ring-teal disabled:opacity-50"
        />
      )}
      {error && <p className="text-xs text-sunset">{error}</p>}
      {maxLength && (
        <p className="text-right text-xs text-dune">{value.length}/{maxLength}</p>
      )}
      <div className="flex items-center gap-3">
        <button
          onClick={handleSubmit}
          disabled={disabled}
          className="rounded-lg bg-teal px-4 py-2 text-sm text-white disabled:opacity-50"
        >
          {submitLabel ?? 'Continue'}
        </button>
        {skipLabel && onSkip && (
          <button
            onClick={onSkip}
            disabled={disabled}
            className="text-xs text-dune hover:text-charcoal disabled:opacity-50"
          >
            {skipLabel}
          </button>
        )}
      </div>
    </div>
  );
}
```

```tsx
    apply() { return Promise.resolve(); },

      if (prop === 'useMutation') {
```

## Notes / Interpreted Decisions

- **[INTERPRETED: `generating_agent`]** — A real `ChatStep` set synchronously in `handleDescriptionSubmit` before the async mutation fires, so typing indicator appears immediately without waiting for a loading state from tRPC.

- **[INTERPRETED: `ask_website` skip path always saves step 6]** — `onSkip` calls `handleWebsiteSubmit('')` so step 6 is persisted before transitioning to `ask_channel`, whether the user submits a URL or skips. Verified: step 6 is the max allowed by `saveStep` Zod schema (`onboarding.ts:443`).

- **[INTERPRETED: URL queue error does not block flow]** — `queueUrl.mutate` has `onError: () => setChatStep('ask_channel')` so a failed URL scrape queue never traps the user. The step 6 save fires fire-and-forget before the mutation so the checkpoint is persisted regardless.

- **[INTERPRETED: `completeOnb` flow]** — `complete()` transitions to `done` state (not directly to `/dashboard`), so the user sees the final message. Navigation happens only when the user explicitly clicks "Open Dashboard".

- **[INTERPRETED: `done` state reachability]** — No code path calls `setChatStep('done')` directly. This guarantees `onboardingComplete = true` is written in the DB before the user ever sees the done screen.

- **[INTERPRETED: `useQuery` mock ignores `enabled`]** — The `buildNestedProxy` mock always returns `queryDataMap` data regardless of `enabled`. This is consistent with `step4-connect-channel.test.tsx` and allows resume tests to inject status data without needing a live Clerk org.

- **[AMBIGUOUS: `status.data.tenantName`]** — `getStatus` returns `{ settings, tenantName, previewCustomerId }` (verified `onboarding.ts:431–435`). `tenantName` can be `null` if the tenant row is missing; handled by `if (status.data.tenantName) setCompanyName(...)`.

- **[Fix 6]** — `'use client'` is the first line of `ChatOnboarding.tsx`. Without it, Next.js 15 App Router rejects hook usage in what it treats as a Server Component.

- **[Fix 7]** — `'use client'` and `import { useState } from 'react'` are the first two lines of `ChatInput.tsx`. Both are required.

- **[Fix 8]** — `ChatBubble.tsx` intentionally omits `'use client'` — it has no hooks and works in both server and client trees.

- **[Fix 9]** — `widgetSnippet` uses `process.env.NEXT_PUBLIC_WIDGET_URL ?? 'http://localhost:5173'` (verbatim match of `Step4ConnectChannel.tsx:27`). No `tenantId` state, no `data-tenant-id` attribute. Webchat UI condition is `channelChoice === 'webchat'` only.

- **[Fix 10]** — WhatsApp section includes `trpc.channel.webhookConfig.useQuery()`, a `Skeleton` import, `handleCopyValue`, and read-only webhook URL + verify token fields with loading skeletons — matching `Step4ConnectChannel.tsx:25,157–206` exactly. The `webhookConfig` query is enabled with the same guard as `getStatus`: `enabled: isLoaded && !!organization`.
